核心能力（从 Matt 移植）

  • Sharpen fuzzy language —— 模糊用语 → 规范术语
  • Discuss concrete scenarios —— 用边界场景检验领域关系
  • Cross-reference with code —— 代码不匹配口头定义时标记
  • Update CONTEXT.md inline —— 术语确定即写，不批量
  • Offer ADRs sparingly —— 仅“难以逆转 + 不记录会困惑 + 真实权衡”三项全满足时

⚠ 路径差异 vs Matt 原版

.claude/skills/domain-modeling/CONTEXT-FORMAT.md

定义 CONTEXT.md 格式——纯 glossary，不含实现细节。

## <Term>
- **Definition**: precise, one sentence
- **Synonyms**: aliases used in code
- **Not to be confused with**: similar but different terms
- **See also**: related terms

## Relationships
- A <Term1> has many <Term2>

## Flagged ambiguities
- <term> was previously used to mean... resolved: ...

.claude/skills/domain-modeling/ADR-FORMAT.md

定义 ADR 格式——仅当"难以逆转 + 不记录会困惑 + 真实权衡"三项全满足时创建。

# ADR-NNNN: <title>

## Status
Proposed | Accepted | Deprecated | Superseded

## Context
What problem are we solving? What constraints?

## Decision
What did we decide? What was the alternative?

## Consequences
What becomes easier? Harder? What is the
migration path?

两份 Builder 做对称修改。以下以 Backend Builder 为例展示，Frontend Builder 同步应用。

Workflow 重构对比

BEFORE — 4 步

Step 1: Read and Understand
Step 2: Plan Your Implementation
Step 3: Implement
"写所有代码 → 写所有测试"（水平切片）
Step 4: Verify (typecheck/lint/tests)

AFTER — 4 步重构

Step 1: Read + CONTEXT.md glossary
Step 2: Plan Test Seams
产出"行为清单 + 接缝 + 优先级"
Step 3: TDD Loop
RED → GREEN → REFACTOR (逐切片)
Step 4: Verify (全量回归)

Step 3：TDD 循环详设（硬门禁）

循环结构（替代原"写所有代码再写所有测试"）：

flowchart TD
    SEAMS["Step 2 Output: <br/>行为清单 + 接缝"] --> VERT["Step 3 Output: <br/>按切片输出 RED→GREEN→REFACTOR 日志"]
    subgraph VERT
      direction TB
      L1["Slice 1: 写测试 → 失败 → 最小代码 → 通过 → 重构"] --> L2["Slice 2: 写测试 → 失败 → 最小代码 → 通过 → 重构"]
      L2 --> L3["Slice 3: ...继续直到所有 behavior 覆盖"]
    end
    VERT --> VERIFY["Step 4: 全量回归"]

禁止模式（水平切片）

RED:   test1, test2, test3, test4
GREEN: impl1, impl2, impl3, impl4
→ 测试的是想象的行为，不是实际行为

要求模式（垂直切片）

RED→GREEN→REFACTOR: test1→impl1
RED→GREEN→REFACTOR: test2→impl2
→ 每个测试响应上一个循环的发现

每循环检查清单（硬门禁）

[ ] 测试描述行为，不描述实现
[ ] 测试仅用公共接口
[ ] 测试能在内部重构后存活（改名不导致失败）
[ ] 代码仅满足当前测试——不写预期功能
[ ] 当前所有测试绿了才进下一个切片

输出格式新增 "TDD Cycle Log" 段落

### TDD Cycle Log
| # | Behavior Tested | Test File | Result | Refactored? |
|---|-----------------|-----------|--------|-------------|
| 1 | User can checkout with valid cart | checkout.test.ts | ✅ RED→GREEN | ✅ extracted discount fn |
| 2 | Empty cart returns error | checkout.test.ts | ✅ RED→GREEN | — |
| 3 | ... | | | |

### Glossary Terms Used
- Order, Checkout, Discount (from CONTEXT.md)
- [any new terms proposed during implementation → Planner should review]

⚠ 与 Planner 的反馈闭环

如果 Builder 在实现中发现需要新领域术语（CONTEXT.md 中不存在），在 "Glossary Terms Used" 段落中标记为 NEW。Planner 在下一轮修订时将这些术语正式化到 CONTEXT.md。这是 下游反馈上游 的闭环。

BEFORE — Phase 1

Phase 1: Reproduce the Failure in Your Mind
1. Clarify the expected behavior
2. Clarify the actual behavior
3. Identify the gap

→ 纯心智模型，无强制输出

AFTER — Phase 1（融合反馈环）

Phase 1: Build Feedback Loop + Reproduce

1. Construct a pass/fail signal for THIS bug:
   - 按优先级尝试：failing test → curl script → CLI invocation → headless browser → replay trace → throwaway harness → fuzz loop → bisection
2. The loop MUST be:
   □ Red-capable (asserts user's EXACT symptom)
   □ Deterministic (same verdict every run)
   □ Fast (seconds, not minutes)
   □ Agent-runnable (you can invoke it unattended)
3. 输出格式：
   **Command**: `[paste the exact invocation]`
   **Output**: `[paste the output showing RED]`
   → 没有此输出，不进入 Phase 2
💡

软指引——不设硬门禁

如果 Debugger 经过"按优先级尝试"后说明"无法在无人工干预下建反馈环"，列出尝试过的方案，并请求用户提供 (a) 环境访问、(b) 捕获制品 (HAR/log/core dump) 或 (c) 临时生产插桩授权。然后可直接进入 Phase 2 代码分析。调试环境差异巨大，硬门禁会导致阻塞。

11. 风险评估与缓解

# 风险 概率 影响 缓解措施
R1 TDD 循环增加 Builder 耗时，maxTurns=20 可能不够 中 中 TDD 循环中每个切片 3 turns (RED/GREEN/REFACTOR)，假设 5 slices = 15 turns + 5 turns 开销 = 20。首次运行后监控，若频繁超限则上调到 25

12. 核心设计决策记录

DD-1：接入范围限定 Phase 1

只接入 domain-modeling、TDD、diagnosing-bugs Phase 1。grilling 交互式追问、codebase-design 深度模块、prototype 原型保留为 Phase 2 候选。理由：改动面小、收益明确、一个下午可完成。

DD-2：混合门禁

domain-modeling (glossary 产出) 和 TDD (RED→GREEN→REFACTOR) 设硬门禁——结构化的、可检查的。diagnosing-bugs 反馈环设软指引——调试环境差异巨大，硬门禁会导致阻塞。

DD-3：domain-modeling 独立文件 + 其余内化

glossary 是跨 Agent 共享的基础设施——独立成 skill 文件。TDD 和调试流程是特定 Agent 的内部纪律——直接改写 Agent 定义，更简洁、token 更省。

DD-4：CONTEXT.md 放在 .claude/context/ 下

与 Matt 原版（项目根目录）不同，FF 包放在 .claude/ 下保持自包含、可移植。smoke test 已校验 .claude/ 内交叉引用。

DD-5：Builder Step 2 + Step 3 嵌入替换

保留 Step 1 (Read) 和 Step 4 (Verify)，Step 2 改造成"Plan Test Seams"，Step 3 改造成"TDD Loop"。保留独立的 Step 4 全量回归保险。

DD-6：Debugger Phase 1 融合而非新增

将"Build feedback loop"要求融入现有 Reproduce 阶段的完成标准——不增加阶段数，不增加 maxTurns 压力，但实质性提升质量门槛。

DD-7：Validator 扩展 2 项——硬门禁检查

领域术语一致性和 TDD 可追溯性是本次接入的核心收益点，不在 Validator 中验证则无法形成闭环。两项都是可机器核查的。

DD-8：Glossary 内嵌蓝图 + 独立文件双写

蓝图内嵌确保下游 Builder 直接读取（蓝图是唯一输入源），独立文件实现跨功能累积。Planner 最终步骤更新 CONTEXT.md。
